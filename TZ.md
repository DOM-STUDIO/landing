# ТЗ на создание статичного landing website для DOM STUDIO

## 1. Контекст проекта

DOM STUDIO — это мастерская/студия handmade-изделий из дерева и смежных материалов с упором на:

- подарки ручной работы;
- персонализацию и гравировку;
- небольшие предметы интерьера;
- отдельные мебельные и декоративные изделия;
- заказы для частных клиентов и для компаний.

По переписке и вложениям сайт нужен не как интернет-магазин, а как презентационный landing / сайт-визитка, который можно отправлять клиентам и компаниям, чтобы быстро показать уровень работ, стиль бренда и способы связи.

Формат первого релиза: статичный сайт на `HTML/CSS/JS` без backend и без CMS.

---

## 2. Главная цель сайта

Сайт должен за 10-20 секунд дать посетителю понятный ответ на 4 вопроса:

1. Кто такие DOM STUDIO.
2. Что именно они делают.
3. Как выглядит качество их работ.
4. Как быстро с ними связаться и обсудить заказ.

Основная конверсия первого релиза:

- переход в Instagram;
- звонок;
- письмо на email;
- переход к личному обсуждению заказа;
- доверительное впечатление у B2C и B2B-аудитории.

---

## 3. Формат продукта

Нужно сделать:

- статичный bilingual landing website;
- две полноценные языковые версии: `RU` и `ET`;
- одна страница на язык;
- публикацию сайта по версиям: `v1`, `v2`, `v3`, `vX`;
- общий визуальный стиль и общая логика структуры;
- без корзины, без оплаты, без личного кабинета, без каталога с backend.

Не делать в первом релизе:

- интернет-магазин;
- checkout;
- фильтрацию каталога с данными из CMS;
- формы, требующие серверной обработки;
- фейковые отзывы, фейковые кейсы, фейковые цифры.

## 4. Целевая аудитория

### 4.1 Частные клиенты

Люди, которые ищут:

- красивый подарок ручной работы;
- персонализированное изделие;
- деревянные изделия для дома;
- подарок на праздник, новоселье, день рождения, свадьбу и т.д.

### 4.2 Компании и серьёзные заказчики

Компании и предприниматели, которым нужны:

- подарки для клиентов или сотрудников;
- брендированные изделия;
- таблички, номера, декоративные элементы;
- нестандартные индивидуальные заказы.

---

## 5. Ключевая идея и позиционирование

Сайт должен транслировать не образ “дешёвого каталога изделий”, а образ:

- аккуратной современной мастерской;
- ручной работы с характером;
- качественного визуального вкуса;
- персонального подхода;
- изделий, которые можно подарить, поставить в интерьер или заказать под задачу.

Желаемое впечатление:

- современно;
- тепло;
- премиально, но не пафосно;
- ремесленно, но не “деревенски”;
- визуально чисто;
- внушает доверие.

## 6. Языки и структура локализации

### 6.1 Языки

Обязательные языки:

- русский;
- эстонский.

Обе версии должны быть полноценными. Нельзя делать “основной язык + сырой дубль”.

### 6.2 Структура директорий

Сайт должен лежать в папке:

- `website/`

Языковая структура:

```text
website/
  index.html
  v1/
    index.html
    ru/
      index.html
    et/
      index.html
    assets/
      css/
      js/
      img/
      fonts/
  v2/
    index.html
    ru/
      index.html
    et/
      index.html
    assets/
      css/
      js/
      img/
      fonts/
```

### 6.3 Решение по `index.html` в корне сайта

Не использовать символическую ссылку.

Вместо этого сделать обычный файл:

- `website/index.html`

Этот файл должен быть корневой страницей со ссылками на опубликованные версии сайта, а не redirect-entrypoint на язык по умолчанию.

Он должен:

- показывать ссылки на доступные версии: `./v1/`, `./v2/`, `./v3/`, `./vX/`;
- быть простым техническим HTML-файлом без сложного дизайна;
- позволять открыть нужную версию вручную;
- не зависеть от JavaScript.

Почему так:

- это даёт прозрачную историю итераций;
- это упрощает сравнение версий, показ заказчику и откат;
- это надёжно работает на GitHub Pages;
- это понятнее в ручном сопровождении.

Правило сопровождения:

- каждое заметное следующее изменение делается в новой папке версии;
- предыдущие версии не перезаписываются без отдельной причины;
- рабочая схема: `v1` -> копия в `v2` -> правки в `v2` -> копия в `v3` и так далее.

### 6.4 Корень каждой версии

Внутри каждой версии должен быть свой корневой файл:

- `website/v1/index.html`
- `website/v2/index.html`
- `website/vX/index.html`

Именно он является redirect-entrypoint на язык по умолчанию внутри конкретной версии.

Требования к version-root файлу:

- должен работать как с JavaScript, так и без него;
- должен содержать fallback-ссылки на `./et/` и `./ru/`;
- должен быть коротким и техническим;
- должен вести только внутрь своей версии.

### 6.5 Переключение языка на сайте

На каждой языковой версии должен быть явный переключатель языка:

- на `ru`-странице ссылка на `../et/`;
- на `et`-странице ссылка на `../ru/`.

Требования:

- переключатель видим в первом экране;
- не прятать его глубоко в меню;
- активный язык визуально выделен;
- переключение мгновенное, без автоопределения языка браузера.

Дополнительно:

- структура секций в `RU` и `ET` версиях должна быть одинаковой;
- якоря секций должны быть синхронизированы по смыслу;
- это упростит поддержку, навигацию и дальнейшие изменения.

## 7. Технические ограничения и базовые требования

### 7.1 Стек

Только:

- HTML;
- CSS;
- vanilla JavaScript.

Без обязательного сборщика.

Допускается:

- общая папка `assets/`;
- общие CSS/JS для обеих языковых версий;
- простая ручная структура, которую легко редактировать без фреймворков.

### 7.2 Подход к путям

Все пути к ресурсам должны быть относительными, а не корневыми.

Нельзя использовать:

- `/assets/...`

Нужно использовать относительные пути, например:

- `../assets/...`
- `./assets/...`
- `./v1/`
- `./v2/`

Это важно для корректной работы на GitHub Pages, особенно если сайт будет открываться не из доменного корня, а из пути репозитория.

Дополнительно:

- языковые страницы внутри `website/vX/ru/` и `website/vX/et/` должны ссылаться только на ассеты своей версии;
- root-файл конкретной версии должен ссылаться только на `./ru/` и `./et/` внутри этой же версии;
- корневой `website/index.html` должен ссылаться только на папки версий.

### 7.3 Безопасность и простота

Первый релиз не должен требовать:

- серверного окружения;
- базы данных;
- npm/yarn/pnpm как обязательной части запуска;
- внешней панели управления.

## 8. Бренд и исходные данные

### 8.1 Название бренда

Основное имя:

- `DOM STUDIO`

### 8.2 Теглайн

Из бренд-материалов подтверждён слоган:

- `Eesti käsitöö kingiideed`

Важно:

- это часть визуальной айдентики;
- при необходимости может использоваться как часть логотипного блока;
- основной текст на страницах при этом должен быть локализован под RU и ET.

### 8.3 Подтверждённые контакты из материалов

Подтверждённые данные:

- телефон: `+372 57 818 661`
- email: `homepagedom@gmail.com`
- Instagram: `https://www.instagram.com/homedompage/`
- Facebook: в материалах есть указание `Макс Домашкевич`, но точный публичный URL не подтверждён

Требование:

- телефон, email и Instagram считать обязательными контактами для первого релиза;

Логотип есть в предоставленных PDF и изображениях.

Требование для первого релиза:

- использовать качественно экспортированную версию логотипа из имеющихся материалов;
- предусмотреть лёгкую замену на `SVG` в будущем без переделки структуры сайта.

## 9. Источники контента и приоритет медиа

### 9.1 Высокий приоритет

Основу визуала должны составлять чистые и сильные product-фото:

- белофонные предметные фотографии;
- крупные кадры досок, номерков, гравировки, мебели и интерьерных изделий;
- логотипные материалы;
- бренд-баннеры и визитка как источник фирменной идентики, а не как контент для прямой вставки на сайт.

### 9.2 Средний приоритет

Можно использовать как вторичный портфель:

- мебельные изделия;
- декоративные панели;
- epoxy / river-style изделия;
- интерьерные предметы, если они визуально усиливают образ мастерской.

### 9.3 Низкий приоритет или исключить

Не использовать как контент сайта:

- технические или случайные изображения, не относящиеся к DOM STUDIO;
- пример изображения, которое показывало формат отправки файла;
- слабые, шумные или визуально устаревшие кадры, если они портят общий уровень подачи.

Отдельно:

- изображение `photo_2025-11-18_21-42-18.jpg` не относится к содержанию сайта и не должно использоваться на лендинге.

### 9.4 Принцип отбора портфолио

В итоговом сайте должны победить не “все работы подряд”, а только сильные и репрезентативные.

Правило:

- лучше меньше, но лучше;
- лучше 10-16 сильных изображений, чем 30 слабых;
- на первом экране и в ключевых секциях использовать только лучшие работы.

## 10. Общая структура сайта

Каждая языковая версия — это одна длинная landing-страница.

Рекомендуемая структура:

1. Header.
2. Hero.
3. О студии / About.
4. Направления работ / What we make.
5. Портфолио / Gallery.
6. Персонализация и индивидуальные заказы.
7. Блок для компаний / B2B.
8. Контакты / CTA.
9. Footer.

## 11. Подробная структура секций

### 11.1 Header

Содержимое:

- логотип;
- переключатель языка `RU / ET`;
- якорная навигация по основным блокам;
- кнопка связи или CTA.

Требования:

- header должен быть компактным;
- на мобильных не занимать слишком много высоты;
- допускается sticky/fixed header;
- навигация должна быть понятной и не перегруженной.

### 11.2 Hero

Задача первого экрана:

- мгновенно показать бренд;
- показать, что это handmade / custom / wooden craft brand;
- вызвать доверие;
- повести к просмотру работ или к контакту.

Hero должен содержать:

- название DOM STUDIO;
- краткий сильный подзаголовок;
- 1-2 CTA;
- визуал высокого качества.

Пример смыслового наполнения hero:

- handmade wood gifts and decor;
- custom engraving;
- unique pieces for home, gifts and businesses;
- crafted in Estonia.

Hero не должен быть:

- перегруженным текстом;
- похожим на шаблон интернет-магазина;
- с банальным стоковым изображением;
- с кричащей графикой.

### 11.3 О студии

Краткий блок, который объясняет:

- кто вы;
- что вы создаёте;
- чем отличается ваш подход;
- что возможны индивидуальные заказы и персонализация.

В этом блоке важно передать:

- ручную работу;
- внимание к деталям;
- работу под заказ;
- эстетику изделия, а не просто “производство”.

### 11.4 Направления работ

Нужен блок с 4-6 карточками категорий.

Потенциальные направления:

- подарки и персонализированные изделия;
- разделочные и сервировочные доски;
- декор и интерьерные изделия;
- таблички, номера и лазерная/гравировальная работа;
- небольшая мебель;
- индивидуальные заказы.

Требования:

- карточки должны быть визуально понятными;
- текст короткий;
- акцент на реальных направлениях из материалов;
- не обещать то, что не подтверждено контентом.

### 11.5 Портфолио / Галерея

Это один из главных блоков сайта.

Должен:

- показывать лучшие работы;
- быть крупным и “воздушным”;
- позволять рассмотреть изделия;
- усиливать впечатление качества.

Технические требования:

- адаптивная grid-галерея;
- кликабельные изображения;
- lightbox / enlarged view без тяжёлых библиотек;
- lazy-loading для не-первых изображений;
- alt-тексты;
- корректное управление с клавиатуры.

Галерея должна включать:

- чистые предметные фото;
- часть интерьерных / контекстных фото, если они действительно украшают подачу;
- визуальное разнообразие, но в едином уровне качества.

### 11.6 Персонализация и индивидуальные заказы

Отдельный блок, потому что это важное конкурентное преимущество.

Нужно показать:

- можно адаптировать изделие под клиента;
- можно добавить имя, надпись, номер, символ, гравировку;
- можно обсудить нестандартный заказ.

Смысл блока:

- объяснить, что DOM STUDIO делает не только готовые вещи, но и кастомные решения.

### 11.7 Блок для компаний / B2B

Так как сайт должен работать и на “серьёзных клиентов-фирмы”, нужен специальный блок.

Он должен объяснять, что DOM STUDIO подходит для:

- корпоративных подарков;
- брендированных изделий;
- небольших серий под компанию;
- интерьерных и идентификационных решений;
- нестандартных заказов.

Важно:

- этот блок не должен выглядеть как отдельный корпоративный сайт;
- он должен быть встроен в общий лендинг и говорить: “мы умеем и для бизнеса тоже”.

### 11.8 Контакты и финальный CTA

Финальный блок обязан быть сильным и конверсионным.

В нём должны быть:

- телефон;
- email;
- Instagram;
- при необходимости QR на Instagram;
- понятный призыв написать/позвонить.

Варианты CTA по смыслу:

- обсудить заказ;
- написать в Instagram;
- связаться по телефону;
- заказать индивидуальное изделие.

Если используется QR:

- он должен быть вспомогательным элементом;
- не должен визуально захватывать весь экран;
- должен вести на подтверждённую ссылку Instagram.

### 11.9 Footer

Минимум:

- логотип или название;
- копирайт;
- ссылки на языковые версии;
- основные контакты или сокращённые ссылки.

## 12. Визуальное направление

### 12.1 Общий стиль

Нужен современный, эстетичный и уверенный визуальный стиль.

Направление:

- premium craft;
- modern handmade;
- Scandinavian / Baltic clarity;
- clean editorial landing;
- тёплое, но не “сельское” настроение.

### 12.2 Чего избегать

Не делать:

- шаблонный “marketplace look”;
- визуал в духе дешёвого онлайн-магазина;
- тяжёлые деревянные текстуры на весь экран;
- дешёвые тени, псевдо-3D кнопки, устаревшие эффекты;
- перегруз декоративными элементами;
- фиолетовые / случайные акцентные цвета;
- слишком “рустикальный” вайб, который удешевляет бренд.

### 12.3 Цветовая логика

Базовая палитра должна быть близка к:

- тёплому белому;
- графиту / чёрному;
- мягкому древесному акценту;
- спокойным натуральным оттенкам.

Цвета должны:

- поддерживать фотографии;
- не спорить с изделиями;
- подчёркивать дерево и ручную работу.

### 12.4 Типографика

Шрифты должны:

- поддерживать кириллицу и эстонские символы;
- быть выразительными, но читабельными;
- работать и в крупных заголовках, и в небольших подписях.

Желательно:

- 1 акцентный шрифт для заголовков;
- 1 удобный шрифт для основного текста;
- не скатываться в обычный безликий системный вид.

### 12.5 Использование логотипа

Логотип должен применяться аккуратно:

- не растягивать;
- не ухудшать контраст;
- не использовать слишком маленьким;
- сохранить возможность лёгкой замены растровой версии на векторную позже.

## 13. UX и взаимодействия

Нужны лёгкие, уместные, не навязчивые взаимодействия:

- плавный скролл по якорям;
- аккуратное появление секций;
- hover-состояния;
- lightbox в галерее;
- понятные состояния кнопок и ссылок.

Не нужны:

- тяжёлые слайдеры “ради слайдеров”;
- параллаксы, которые мешают;
- автопроигрывающиеся видео;
- сложные анимации, замедляющие сайт.

---

## 14. Адаптивность

Сайт обязан корректно работать:

- на мобильных;
- на планшетах;
- на ноутбуках;
- на широких десктопах.

Контрольные ширины:

- 320px+
- 375px+
- 768px+
- 1024px+
- 1440px+

Требования:

- header не ломается;
- галерея красиво перестраивается;
- текст остаётся читабельным;
- кнопки достаточно крупные;
- язык и CTA остаются доступными на мобильных.

---

## 15. Доступность

Минимальные требования по accessibility:

- семантическая структура HTML;
- корректный `lang` для каждой страницы;
- alt-тексты для изображений;

## 21. GitHub Pages и деплой

### 21.1 Общий подход

Нужен GitHub Action для автоматического деплоя в GitHub Pages.

Путь публикации:

- содержимое директории `website/`

### 21.2 Почему не symlink

Для GitHub Pages в этом проекте нужно избегать символических ссылок в публикуемом артефакте.

Поэтому:

- `website/index.html` должен быть обычным HTML-файлом со ссылками на версии;
- `website/vX/index.html` должен быть обычным HTML-файлом-редиректом внутри версии;
- не использовать symlink ни для выбора версии, ни для выбора языка.

### 21.3 Workflow

Нужно создать workflow:

- `.github/workflows/deploy-pages.yml`

Требования к workflow:

- запуск по `push` в default branch;
- запуск по `workflow_dispatch`;
- публикация через GitHub Pages Actions;
- деплой только статического содержимого папки `website/`;
- без build-step, если сборка не требуется.

Рекомендуемый стек action-ов:

- `actions/checkout`
- `actions/configure-pages`
- `actions/upload-pages-artifact`
- `actions/deploy-pages`

На момент подготовки этого ТЗ ориентироваться на актуальные версии официальных GitHub Actions для Pages.

### 21.4 Дополнительные требования к деплою

- repository settings должны использовать `GitHub Actions` как source для Pages;
- workflow должен иметь корректные permissions для Pages deployment;
- желательно добавить `concurrency`, чтобы не плодить параллельные деплои;
- публикуемая папка не должна содержать мусор и лишние исходники.
